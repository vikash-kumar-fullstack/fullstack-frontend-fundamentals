INTRODUCTION
--------------------------------------------------------------------------------
Modern web applications must securely manage user identity, access control,
and protection against common web vulnerabilities. When a user interacts with
a web application, the backend must ensure that the user is properly identified
and allowed to perform only authorized actions.

This requires several interconnected mechanisms including authentication,
authorization, session management, token-based authentication, and browser
security policies.

These concepts are fundamental in backend development and are frequently
discussed in technical interviews and system design discussions.

The key concepts covered in this document include:

Authentication
Authorization
Login Flow
Cookies
Sessions
Session Scaling Problems
JWT (JSON Web Token)
JWT Structure
JWT Authentication Flow
CORS (Cross-Origin Resource Sharing)
XSS (Cross-Site Scripting)
CSRF (Cross-Site Request Forgery)

SECTION 1: AUTHENTICATION VS AUTHORIZATION
================================================================================

Authentication and Authorization are two closely related but fundamentally
different concepts in backend systems.

Although they sound similar, they represent different stages of access control.


AUTHENTICATION

Authentication refers to the process of verifying the identity of a user.

In simple terms, authentication answers the question:

"Who are you?"

When a user attempts to log into an application, they provide credentials such
as an email and password. The backend server receives these credentials and
compares them with the data stored in the database.

If the credentials match the stored data, the server confirms the identity of
the user. This entire verification process is known as authentication.

Authentication mechanisms can include:

Username and password
OTP verification
Biometric authentication
Social login systems


AUTHORIZATION

Authorization occurs after authentication and determines what actions the user
is allowed to perform.

Authorization answers the question:

"What are you allowed to do?"

After the server confirms the user’s identity, it checks the user’s permissions
or role within the system.

For example:

An admin user may be allowed to delete accounts or manage system settings.

A regular user may only be allowed to view or edit their own profile.

Therefore:

Authentication verifies identity.
Authorization determines access permissions.

Typical application flow:

User logs in → Authentication occurs → Server verifies identity
↓
Server checks user permissions → Authorization determines allowed actions


================================================================================
SECTION 2: WHAT HAPPENS WHEN A USER LOGS IN
================================================================================

When a user logs into a web application, several steps occur between the client
and the backend server.

The process typically begins when the client sends a login request to the
server containing user credentials.

Example login request:

POST /login

Body:
{
  "email": "user@email.com",
  "password": "password123"
}

The backend server receives this request and verifies the credentials by
checking them against records stored in the database.

If the credentials are valid, the server must create a way to remember the user
for future requests.

This is necessary because HTTP is a stateless protocol.


STATELESS PROTOCOL

A stateless protocol means that each request is independent and the server does
not automatically remember previous interactions.

Because of this, the server must implement a mechanism to maintain user state.

Two common approaches are used:

Session-based authentication
Token-based authentication using JWT

Once authentication is successful, the server sends an identifier back to the
client. The client stores this identifier and sends it with every future
request so the server can recognize the user.


================================================================================
SECTION 3: COOKIES
================================================================================

Cookies play a critical role in maintaining state between client and server.

A cookie is a small piece of data that the server sends to the browser. The
browser stores this data and automatically includes it in future requests to
the same server.

Cookies are stored inside the browser and are attached to requests
automatically.

Example server response header:

Set-Cookie: sessionId=abc123

After receiving this response, the browser stores the cookie internally.

When the browser sends another request to the same server, it includes the
cookie in the request header.

Example request header:

Cookie: sessionId=abc123

This allows the server to recognize the user without requiring the user to log
in again.

Cookies can store information such as:

Session identifiers
Authentication tokens
User preferences
Tracking information

Cookies are essential for maintaining state in stateless protocols like HTTP.


================================================================================
SECTION 4: SESSIONS
================================================================================

Sessions are a traditional method of managing authentication on the server
side.

In session-based authentication, the server stores user information on the
server after the user logs in.

When a user successfully logs in, the server creates a session object that
contains information about the user.

The server then generates a unique session identifier called a session ID.

This session ID is sent to the client inside a cookie.

Example:

Set-Cookie: sessionId=abc123

The client stores this session ID and sends it with every request.

When the server receives a request containing the session ID, it looks up the
corresponding session data stored on the server.

If the session exists and is valid, the server identifies the user and
processes the request.


SESSION AUTHENTICATION FLOW

User logs into application
↓
Server verifies credentials
↓
Server creates session object
↓
Server generates session ID
↓
Session ID stored in cookie
↓
Client sends cookie with every request
↓
Server validates session


Sessions are secure and simple but introduce scalability challenges when
applications grow.


================================================================================
SECTION 5: SESSION SCALING PROBLEM
================================================================================

One major limitation of session-based authentication is scalability.

Because session data is stored on the server, the server must maintain session
records for every active user.

If an application has millions of users, the server must store millions of
sessions.

This can consume large amounts of server memory and degrade performance.

Another issue occurs in distributed systems with multiple servers.

Example scenario:

User logs in through Server A
Session is stored on Server A

If the user's next request is handled by Server B, Server B will not recognize
the session.

This causes authentication failure.

Solutions include:

Shared session storage systems
Examples:

Redis
Memcached

However, these introduce additional infrastructure complexity.

To avoid these problems, many modern applications use JWT-based authentication.


================================================================================
SECTION 6: JWT (JSON WEB TOKEN)
================================================================================

JWT stands for JSON Web Token.

It is a compact and self-contained method for securely transmitting information
between client and server.

JWT uses stateless authentication.

This means the server does not store session data.

Instead, all necessary information about the user is stored inside the token.

When a user logs in successfully, the server generates a JWT and sends it to
the client.

The client stores the token and sends it with every request.

The server verifies the token each time to authenticate the user.

Because the server does not store session data, JWT works well in distributed
systems and microservice architectures.


================================================================================
SECTION 7: JWT STRUCTURE
================================================================================

A JWT consists of three parts separated by dots.

HEADER.PAYLOAD.SIGNATURE


HEADER

The header contains metadata about the token such as the signing algorithm.

Example:

{
 "alg": "HS256",
 "typ": "JWT"
}


PAYLOAD

The payload contains claims about the user.

Claims may include:

User ID
Email
Role
Token expiration time

Example:

{
 "userId": 42,
 "email": "user@email.com",
 "exp": 1730000000
}

Important note:

The payload is NOT encrypted.

It is only encoded using Base64.

Therefore sensitive information should never be stored inside the payload.


SIGNATURE

The signature ensures that the token has not been tampered with.

The signature is generated using:

Encoded Header
Encoded Payload
Secret Key

If someone modifies the payload, the signature verification will fail.


================================================================================
SECTION 8: JWT AUTHENTICATION FLOW
================================================================================

The JWT authentication process works as follows:

User sends login request
↓
Server verifies credentials
↓
Server generates JWT
↓
Token sent to client
↓
Client stores token
↓
Client sends token with future requests
↓
Server verifies token
↓
Access granted


JWT tokens are usually sent in the Authorization header.

Example:

Authorization: Bearer <token>

The server extracts the token and verifies it using the secret key.

If verification succeeds, the server decodes the payload and identifies the
user.

SECTION 9: CORS (CROSS-ORIGIN RESOURCE SHARING)
================================================================================

CORS is a browser security mechanism.

CORS stands for:

Cross-Origin Resource Sharing.

Browsers enforce a policy called the Same Origin Policy.


SAME ORIGIN POLICY (SOP)

This rule states that a webpage can only send requests to the same origin
from which it was loaded.

An origin is defined by three components:

Protocol
Domain
Port

Example:

http://localhost:3000
http://localhost:5000

These are considered different origins because the ports differ.

CORS allows servers to specify which origins are allowed to access their
resources.

Example header:

Access-Control-Allow-Origin: http://localhost:3000


PREFLIGHT REQUEST

Before certain requests, the browser sends an OPTIONS request called a
preflight request.

The purpose is to check whether the server allows the actual request.

SECTION 10: XSS (CROSS-SITE SCRIPTING)
================================================================================

XSS stands for Cross-Site Scripting.

It is a vulnerability that allows attackers to inject malicious scripts into
web pages viewed by other users.

When the victim loads the page, the malicious script executes in their browser.

Possible consequences include:

Stealing cookies
Session hijacking
Redirecting users to malicious websites


TYPES OF XSS

Stored XSS

Malicious scripts are stored in the database and executed whenever users load
the page.


Reflected XSS

Malicious scripts are included in a request and immediately reflected in the
response.


DOM-Based XSS

Occurs when client-side JavaScript manipulates the DOM in unsafe ways.


PREVENTION METHODS

Input sanitization
Output escaping
Content Security Policy
HTTP-only cookies
