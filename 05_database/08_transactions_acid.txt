1. WHAT IS A TRANSACTION?
A transaction is a group of operations executed as a single unit.

Explanation:
A transaction ensures that either all operations are completed successfully or none of them are applied. It prevents partial updates in the database.

Example:
Bank Transfer:
- Deduct 1000 from Account A
- Add 1000 to Account B

Both must succeed together.


------------------------------------------------------------


2. WHY TRANSACTIONS ARE IMPORTANT
Transactions ensure data consistency and reliability in multi-user systems.

Explanation:
Without transactions, if a failure occurs in between operations, the database may go into an inconsistent state.

Example:
Money deducted but not credited → incorrect data


------------------------------------------------------------


3. STATES OF A TRANSACTION

1. Active → Transaction is executing
2. Partially Committed → Last operation executed
3. Committed → Successfully saved
4. Failed → Error occurred
5. Aborted → Rolled back

Explanation:
A transaction moves through these states depending on success or failure.


------------------------------------------------------------


4. COMMIT AND ROLLBACK
-----------------------

COMMIT:
Commit permanently saves the changes made by a transaction.

ROLLBACK:
Rollback cancels all changes and restores previous state.

Example:

START TRANSACTION
Deduct 1000
Add 1000
COMMIT

If error:
ROLLBACK


------------------------------------------------------------


5. WHAT IS ACID PROPERTY?
ACID is a set of properties that ensure reliable transaction processing.

ACID stands for:
- Atomicity
- Consistency
- Isolation
- Durability


------------------------------------------------------------


6. ATOMICITY
Atomicity means a transaction is either fully completed or not executed at all.

Explanation:
No partial execution is allowed. If any step fails, the entire transaction is rolled back.

Example:
Bank Transfer:
If money is deducted but not added → rollback happens

Result:
Either both operations happen or none.


------------------------------------------------------------


7. CONSISTENCY
Consistency ensures that the database remains in a valid state before and after a transaction.

Explanation:
All rules, constraints, and relationships must be maintained.

Example:
If total money in system was 5000 before transaction,
it should remain 5000 after transfer.

Invalid state is not allowed.


------------------------------------------------------------


8. ISOLATION
Isolation ensures that multiple transactions do not interfere with each other.

Explanation:
Each transaction behaves as if it is the only one running.

Example:
Two users booking the same seat:
Only one should succeed.

DBMS ensures no conflict.


------------------------------------------------------------


9. DURABILITY
Durability ensures that once a transaction is committed, it is permanently stored.

Explanation:
Even if system crashes, committed data will not be lost.

Example:
After successful payment,
even if server crashes, order is saved.


------------------------------------------------------------


10. REAL-WORLD EXAMPLE (BANK SYSTEM)

Transaction:
Transfer 1000 from A to B

Step 1: Deduct from A
Step 2: Add to B

ACID Applied:

Atomicity:
Both steps succeed or both fail

Consistency:
Total money remains same

Isolation:
Other users cannot see partial transfer

Durability:
Once done, data is permanently stored


------------------------------------------------------------


11. PROBLEMS WITHOUT ISOLATION

1. Dirty Read
   Reading uncommitted data

2. Non-Repeatable Read
   Same query gives different results

3. Phantom Read
   New rows appear during transaction

Explanation:
Isolation levels control these issues.


------------------------------------------------------------


12. ISOLATION LEVELS (BASIC IDEA)

1. Read Uncommitted → Lowest safety
2. Read Committed → Only committed data visible
3. Repeatable Read → Same data during transaction
4. Serializable → Highest safety

Trade-off:
Higher isolation → Lower performance


------------------------------------------------------------


13. HOW DBMS IMPLEMENTS TRANSACTIONS

- Locking (Row/Table locks)
- Logging (Write-Ahead Logging)
- Concurrency Control
- Recovery Mechanisms

Explanation:
DBMS internally manages transactions using these techniques.


------------------------------------------------------------


14. TRANSACTIONS IN FULL STACK DEVELOPMENT

Example: Online Payment

Flow:
Frontend → Request payment
Backend → Start transaction
DB → Deduct balance + create order
If success → COMMIT
If error → ROLLBACK

Why important:
Prevents data corruption and ensures reliability.
