1. PROJECT STRUCTURE (MONOREPO vs SEPARATION)

---

Structure:
smart-healthcare-system/
backend/
frontend/

Concept:

* Backend and frontend are separated into independent services.
* Each has its own dependency management and runtime.

Why important:

* Independent scaling (backend can scale without frontend)
* Cleaner architecture
* Easier deployment (Docker, CI/CD)
* Industry standard approach

Interview Insight:

* Never mix frontend and backend dependencies in one package.json.
* Microservices and modular systems follow this principle.

2. NODE INITIALIZATION (npm init)

---

Command:
cd backend
npm init -y

Concept:

* Initializes Node.js project
* Creates package.json (dependency + script manager)

Why only in backend:

* Backend is a separate service
* Avoids dependency conflicts
* Enables service-level deployment

Interview Insight:

* package.json defines project metadata, scripts, and dependencies
* Helps in version control and reproducibility

4. ASYNC HANDLER (ERROR WRAPPING)

---

Concept:

* Wrapper function to handle async errors automatically

Problem:

* Repetitive try-catch blocks in controllers

Solution:
const asyncHandler = (fn) =>
(req, res, next) =>
Promise.resolve(fn(req, res, next)).catch(next);

Why important:

* Cleaner code
* Centralized error flow
* Avoids missed error handling

Interview Insight:

* Express does not catch async errors by default
* asyncHandler ensures all errors reach middleware

---

5. CORS (CROSS-ORIGIN RESOURCE SHARING)

---

Concept:

* Browser security mechanism restricting cross-origin requests

Problem:
Frontend (localhost:3000)
Backend (localhost:5000)
→ Blocked by browser

Solution:
app.use(cors())

Advanced:
app.use(cors({
origin: "http://localhost:3000",
credentials: true
}))

Why important:

* Enables frontend-backend communication
* Required for APIs consumed by browsers

Interview Insight:

* "*" allows all origins but is insecure
* credentials=true needed for cookies/auth

7. INPUT VALIDATION (ZOD vs JOI)

---

Problem:

* Invalid user input can break system or corrupt DB

Example:
{ "email": 123 }

Solution:

* Validate before processing

ZOD Example:
const schema = z.object({
email: z.string().email(),
password: z.string().min(6)
});

const result = schema.safeParse(req.body);

Why ZOD:

* Type-safe
* Cleaner syntax
* Modern library

ZOD vs JOI:

* ZOD → modern, TS-friendly
* JOI → older, not type-safe

Why important:

* Prevents invalid data
* Enhances security
* Improves reliability

Interview Insight:

* Validation should happen before business logic
* Can be placed in middleware layer

---

8. ERROR MIDDLEWARE (CENTRALIZED ERROR HANDLING)

---

Concept:

* Single place to handle all errors in application

Implementation:
const errorHandler = (err, req, res, next) => {
res.status(err.status || 500).json({
success: false,
message: err.message || "Server Error"
});
};

Usage:
app.use(errorHandler)

IMPORTANT RULE:

* Must be the LAST middleware

Flow:
Request → Route → Controller → next(err) → Error Middleware → Response

Why important:

* Removes repetitive try-catch
* Consistent error responses
* Centralized debugging

Interview Insight:

* Middleware chain in Express is sequential
* Error middleware has 4 parameters (err, req, res, next)
