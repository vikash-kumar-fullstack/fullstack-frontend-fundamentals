SECTION 4: CORE UTILITIES (ARCHITECTURE BACKBONE)
=================================================

1. API RESPONSE STANDARD

Format:
{
success: boolean,
data: any,
message: string
}

Why:

* Frontend consistency
* API contract clarity

Trade-off:

* Slight abstraction overhead

---

2. ASYNC HANDLER

Purpose:

* Handle async errors automatically

Key Idea:
Express does NOT catch async errors

Implementation:
Promise.resolve(fn(...)).catch(next)

Why:

* Removes try-catch repetition
* Ensures all errors reach middleware

Interview Insight:
“Async handler bridges Express’s limitation with async functions.”

---

3. ERROR MIDDLEWARE

Purpose:

* Centralized error handling

Rule:

* Must be last middleware

Benefits:

* Consistent error response
* Clean controllers

Trade-off:

* Less granular control per route (handled via custom errors)

---

4. LOGGER (MORGAN)

Purpose:

* Logs HTTP requests

Why:

* Debugging
* Monitoring

Production Extension:

* Integrated with ELK / logging systems

---

5. CORS

Purpose:

* Allow cross-origin communication

Key Concept:
Browser security policy

Advanced:

* origin restriction
* credentials support

Security Note:
Avoid using "*" in production

====================================================================
SECTION 5: VALIDATION & SECURITY LAYER
======================================

1. INPUT VALIDATION (ZOD)

Purpose:

* Ensure correct structure and types

Important:

* VALIDATION ≠ SANITIZATION

Zod Role:

* Schema enforcement

Interview Insight:
“Validation ensures data correctness before business logic.”

---

2. SANITIZATION (XSS PROTECTION)

Purpose:

* Remove malicious scripts

Tools:

* xss-clean
* sanitize-html

Important Concept:

* Backend sanitizes input
* Frontend escapes output

Critical Insight:
XSS is mainly a frontend rendering issue

---

3. HELMET (SECURITY HEADERS)

Purpose:

* Adds HTTP security headers

Protects against:

* Clickjacking
* MIME sniffing
* Basic XSS vectors

---

4. PASSWORD SECURITY

Using bcrypt:

* Hashing (one-way)
* Salt rounds (cost factor)

Why:

* Plain passwords must NEVER be stored

Trade-off:

* Higher salt rounds = more security but slower

Interview Insight:
“bcrypt protects against rainbow table attacks using salting.”

---

1. USER MODEL (DATABASE DESIGN)

Fields:

* name
* email (unique)
* phone (unique)
* password
* role (RBAC)
* isActive
* timestamps

Concepts:

a) UNIQUE CONSTRAINT

* Prevent duplicate users

b) RBAC (Role-Based Access Control)

* patient / doctor / admin

c) SOFT CONTROL (isActive)

* Disable user without deleting

d) TIMESTAMPS

* Auditing and tracking

Interview Insight:
“I designed schema with RBAC and soft deletion capability.”

---

2. SERVICE LAYER (BUSINESS LOGIC)

registerUser()

Steps:

1. Check existing user
   → Prevent duplicates

2. Hash password
   → bcrypt.hash(password, 10)

3. Create user

Why Service Layer?

* Keeps controller thin
* Reusable logic
* Better testing

Interview Insight:
“Business logic is isolated in services to maintain clean architecture.”

SECTION 8: INTERVIEW CRITICAL QUESTIONS
=======================================

Q1: Why use service layer?
A:
Separation of concerns, scalability, testability

Q2: Why bcrypt?
A:
Hashing + salting → prevents password attacks

Q3: Does Zod prevent XSS?
A:
No, it validates structure only

Q4: Why asyncHandler?
A:
Express doesn’t handle async errors natively

Q5: Why modular structure?
A:
Improves maintainability and scalability

Q6: How to scale this system?
A:

* Split modules into microservices
* Add caching (Redis)
* Load balancing
