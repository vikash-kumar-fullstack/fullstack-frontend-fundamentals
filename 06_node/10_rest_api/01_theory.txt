1. WHAT IS AN API?
-----------------------------------------------------------

API stands for:

Application Programming Interface

An API is a set of rules that allows two software systems
to communicate with each other.

Example:

Frontend → requests data → Backend API

Example flow:

React App
     |
     | HTTP Request
     |
Backend Server (Node / Express)
     |
     | Database Query
     |
Database


Example API:

GET /users

Response:

[
 {id:1,name:"Vikash"},
 {id:2,name:"Rahul"}
]


-----------------------------------------------------------


2. WHAT IS REST?
-----------------------------------------------------------

REST stands for:

Representational State Transfer

REST is an architectural style used for designing APIs
over HTTP.


REST was introduced by:

Roy Fielding in 2000


Important idea:

REST uses HTTP protocols to manage resources.


-----------------------------------------------------------


3. WHAT IS A RESOURCE?
-----------------------------------------------------------

In REST everything is treated as a resource.

Examples:

User
Product
Order
Cart


Each resource is represented by a URI.

Example:

/users
/products
/orders


-----------------------------------------------------------


4. WHAT IS A RESTFUL API?
-----------------------------------------------------------

A RESTful API is an API that follows the REST architectural
principles.

Example:

GET /users
POST /users
GET /users/10
PUT /users/10
DELETE /users/10


These operations correspond to CRUD operations.


-----------------------------------------------------------


5. CRUD OPERATIONS IN REST
-----------------------------------------------------------

CRUD means:

Create
Read
Update
Delete


Mapping with HTTP methods:


CREATE → POST

POST /users


READ → GET

GET /users


UPDATE → PUT or PATCH

PUT /users/10


DELETE → DELETE

DELETE /users/10


-----------------------------------------------------------


6. REST ARCHITECTURAL CONSTRAINTS
-----------------------------------------------------------

REST APIs follow 6 principles.


-----------------------------------------------------------
1. CLIENT SERVER ARCHITECTURE
-----------------------------------------------------------

Client and server are separate systems.

Client:

Browser
Mobile app
Frontend application

Server:

API server
Database server


This separation allows independent development.


-----------------------------------------------------------
2. STATELESS
-----------------------------------------------------------

Each request must contain all required information.

Server does not store client session.


Example:

Bad example:

Server stores login session.


Good example:

Client sends authentication token with every request.


Example:

Authorization: Bearer token


-----------------------------------------------------------
3. CACHEABLE
-----------------------------------------------------------

Responses should be cacheable when possible.

Caching improves performance.

Example:

GET /products


Browser can cache response.


-----------------------------------------------------------
4. UNIFORM INTERFACE
-----------------------------------------------------------

Uniform interface means standardized communication.

Rules include:

Resource identification using URI

Use of HTTP methods

Standard response format (usually JSON)


-----------------------------------------------------------
5. LAYERED SYSTEM
-----------------------------------------------------------

Client should not know whether it is talking
directly to server or through intermediaries.

Example:

Client → CDN → Load Balancer → API Server


-----------------------------------------------------------
6. CODE ON DEMAND (OPTIONAL)
-----------------------------------------------------------

Server can send executable code to client.

Example:

JavaScript.


-----------------------------------------------------------


7. HTTP METHODS USED IN REST
-----------------------------------------------------------

GET

Retrieve data


POST

Create new resource


PUT

Update entire resource


PATCH

Update partial resource


DELETE

Remove resource


-----------------------------------------------------------


8. IDEMPOTENCY (IMPORTANT INTERVIEW QUESTION)
-----------------------------------------------------------

Idempotent means:

Performing the same request multiple times
produces the same result.


Example:

GET /users

Safe and idempotent.



DELETE /users/10

Calling multiple times still results in user deleted.


POST is NOT idempotent.


-----------------------------------------------------------


9. REST API URL DESIGN
-----------------------------------------------------------

Good REST design uses nouns not verbs.

Bad design:

/getUsers
/createUser


Good design:

/users
/users/10


-----------------------------------------------------------


10. STATUS CODES IN REST API
-----------------------------------------------------------

200 OK

Request successful


201 Created

New resource created


204 No Content

Success but no body returned


400 Bad Request

Invalid client request


401 Unauthorized

Authentication required


403 Forbidden

Access denied


404 Not Found

Resource does not exist


500 Internal Server Error


-----------------------------------------------------------


11. REQUEST BODY IN REST
-----------------------------------------------------------

POST and PUT requests send body data.

Example JSON body:

{
 name: "Vikash",
 email: "vikash@gmail.com"
}


-----------------------------------------------------------


12. REST RESPONSE FORMAT
-----------------------------------------------------------

REST APIs commonly return JSON.

Example:

{
 id:1,
 name:"Vikash",
 role:"Developer"
}


-----------------------------------------------------------


13. PAGINATION IN REST API
-----------------------------------------------------------

Large data sets should be paginated.

Example:

GET /users?page=1&limit=10


-----------------------------------------------------------


14. VERSIONING IN REST API
-----------------------------------------------------------

APIs evolve over time.

Versioning prevents breaking old clients.


Example:

/api/v1/users
/api/v2/users
